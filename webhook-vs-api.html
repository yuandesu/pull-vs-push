Polling vs Webhook

兩種透過 API 取得資料的模式
從基本概念到 Datadog 實戰範例

PULL — 主動輪詢 PUSH — 事件推送

⚠️ Webhook 不是 API 的相反。Webhook 本身就是透過 HTTP API 實現的。
真正的差別是「誰主動發起 request」。

按右方向鍵 → 或下方按鈕開始

先搞清楚一件事

Webhook ≠ API 的反義詞

API 是一套介面規範，定義了系統之間如何溝通（透過 HTTP request/response）。
Webhook 是一種設計模式，讓系統在事件發生時，自動呼叫你預先註冊的 URL。

Webhook 的實作方式就是發送一個 HTTP POST request — 它本身就是透過 API 來實現的。
所以真正該比較的是：Polling（輪詢） vs Webhook（事件推送），兩者都使用 API。

🔄 Polling（輪詢）

「你主動、重複地去問」

你的程式定期呼叫 API 檢查有沒有新資料。
像你每 5 分鐘打電話給餐廳問：「外送到了嗎？」

PULL 模式 · 你發起 API call

🪝 Webhook（事件推送）

「對方主動通知你」

你提供一個 endpoint URL，對方在事件發生時自動 POST 過來。
像外送 App 主動推播：「餐點已送達」

PUSH 模式 · 對方發起 API call

📌 Pull 模式還可以再細分：

ON-DEMAND（按需呼叫）

需要資料時才呼叫，不固定頻率。
例：使用者點擊搜尋 → 發一次 REST API request

POLLING（定期輪詢）

程式自動、固定間隔反覆呼叫，不管有沒有新資料。
例：cron job 每 5 分鐘 call 一次 API 確認狀態

⚠️ Pull 的主要缺點來自 Polling（定期呼叫）：大量重複 request、多數回應沒有變化、浪費資源。On-demand 的一次性呼叫不會有這個問題。

詳細比較

Polling vs Webhook 差異一覽

面向🔄 Polling（主動輪詢 API）🪝 Webhook（事件推送）
誰發起 request你的程式主動 call API對方系統主動 POST 到你的 endpoint
通訊方向你 → 對方 API（Request/Response）對方 → 你的 API endpoint（單向推送）
即時性取決於 polling 頻率（有延遲）事件發生即時通知
資源消耗反覆 polling 浪費資源（大部分回應沒變化）只在有事件時才有流量
你需要提供API Key / Token（呼叫對方 API）一個公開可接收的 HTTP endpoint
錯誤處理直接看 response status code需自建 retry / 冪等機制
底層實作HTTP API call（GET / POST）HTTP API call（POST）— 一樣是 API

📌 最後一行是重點：兩者的底層技術都是 HTTP API call，只是主動權不同。

Datadog 實戰

在 Datadog 裡的兩種模式

Datadog 同時提供 API（讓你主動查詢/操作）和 Webhook（讓 Datadog 主動通知你）。

呼叫 Datadog API（Polling）

你主動向 Datadog 發 request

  • 查詢 monitor 狀態
  • 建立 / 更新 dashboard
  • 取得 metrics 資料
  • 管理 downtime
  • 提交自定義 event
# 你 → Datadog API
GET api.datadoghq.com/api/v1/monitor/12345
# Datadog → 你（response）
{ "status": "Alert", "name": "CPU High" }

Datadog Webhook（事件推送）

Datadog 主動 POST 到你的 endpoint

  • Monitor 觸發 alert 時自動通知
  • 推送告警到 Slack / PagerDuty / 自建系統
  • 觸發自動化修復流程
  • 同步告警到第三方 ITSM
  • 整合 ChatOps 工作流
# Datadog → 你的 endpoint（也是一個 API）
POST your-server.com/webhook/datadog
# Payload（自動推送）
{ "alert_type": "error", "title": "CPU > 90%" }

使用場景

什麼時候該用哪個？

Polling API 的場景

  • 定時產生報表（每日 monitor 狀態彙整）
  • 批次操作（建立 / 更新多個 monitor）
  • 查詢歷史 metrics 做分析
  • Terraform / IaC 管理 Datadog 設定
  • 需要 on-demand 取得特定資料

Webhook 推送的場景

  • Alert 即時通知（Slack / Teams / PagerDuty）
  • 自動建立 incident ticket（Jira / ServiceNow）
  • 觸發自動修復腳本（restart service）
  • 同步告警狀態到內部系統
  • ChatOps 工作流整合

💡 實務上兩者經常搭配使用：
Webhook 即時通知 →「收到 alert！」→ 用 API 查詢更多 context →「影響了哪些 host？」

總結

Polling = 你主動 call API  |  Webhook = 對方 call 你的 API

兩者底層都是 HTTP API call，差別只在誰主動發起 request。

🔄 Polling🪝 Webhook
主動方你（Client）對方（Server）
觸發你的 request / 排程事件發生
你需要對方的 API Key你自己的 Endpoint URL
即時性⚠️ 取決於 polling 頻率✅ 即時
底層技術HTTP APIHTTP API（一樣的）

在 Datadog 中：Polling API 查資料、管設定  /  Webhook 接收即時告警通知
兩者經常搭配：Webhook 即時通知 → 再用 API 查更多細節
